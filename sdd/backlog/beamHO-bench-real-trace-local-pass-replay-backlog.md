# beamHO-bench — Real-Trace Local-Pass Fixture and Replay Window SDD

**Version:** 0.1.0
**Date:** 2026-03-16
**Status:** Backlog Draft / Not Active Pending

---

## 1. Purpose

Define a research-grounded redesign of the `real-trace` input and startup policy so observer-sky rendering uses a physically meaningful local-pass pool instead of a constellation-wide even sample that can under-represent high-elevation NTPU passes.

This package exists because the current `real-trace` path has three distinct concerns that are not yet modeled separately:
1. epoch correctness:
   - runtime should replay from the fixture epoch, not from wall-clock now.
2. fixture relevance:
   - a constellation-wide even sample does not preserve observer-local pass density.
3. presentation mode:
   - a deterministic replay window suitable for frontend/demo review is not the same contract as research-default continuous propagation.

The package also defines how startup bootstrap and optional replay looping should behave without introducing fake satellites or non-physical motion.

---

## 2. Scope Boundary

In scope:
1. define the `real-trace` time-origin contract explicitly.
2. replace constellation-wide even sampling with observer-local pass selection for fixture generation.
3. define deterministic startup bootstrap so the initial frame can begin at a locally pass-rich epoch inside the selected replay window.
4. define an optional non-default replay-window loop mode for frontend/demo inspection.
5. define validation gates for epoch correctness, local-pass coverage, startup visibility, and loop-seam behavior.

Out of scope:
1. changing handover KPI formulas, scheduler logic, or beam scheduling policy.
2. inventing synthetic satellites, fake azimuth offsets, or non-physical display randomization.
3. changing the fixed NTPU observer coordinate.
4. replacing SGP4 / Kepler propagation math itself.
5. promoting this package to active pending or implementation work in the same change set.

---

## 3. Problem Statement

The current `real-trace` workflow correctly replays from fixture epoch, but the fixture itself is generated by taking an even provider-wide sample with a fixed record cap. That policy is frontend-friendly in size, but it is not observer-aware.

Observed consequence:
1. the visible local-pass density over NTPU can be greatly reduced relative to the full source snapshot.
2. startup at replay time `t=0` may contain few or no high-elevation satellites even when the full constellation snapshot would contain many.
3. the resulting screen can misleadingly read as "Starlink over NTPU is sparse" when the real issue is the sampling policy, not the propagation or projection math.

This package separates:
1. source freshness
2. replay epoch correctness
3. observer-local fixture selection
4. optional demo-loop behavior

## 4. Normative Requirements (MUST/SHALL)

### 4.1 Epoch Contract

1. `real-trace` SHALL use fixture-defined epoch metadata as simulation time origin.
2. `real-trace` SHALL NOT derive its default start time from wall-clock now.
3. Fixture refresh cadence SHALL be treated as data-freshness governance, not as runtime time-origin logic.
4. Fixture metadata SHALL record:
   - source snapshot file
   - source snapshot generation time when available
   - replay window start epoch
   - replay window duration or end epoch
5. The startup bootstrap selected for rendering SHALL remain inside the fixture replay window and SHALL be deterministic for the same fixture and config.

### 4.2 Observer-Local Pass Fixture Policy

1. Fixture generation for observer-sky `real-trace` SHALL begin from the full provider snapshot available under `tle_data`.
2. The builder SHALL propagate candidate satellites across an observer-specific analysis window instead of sampling uniformly across the entire constellation first.
3. A satellite SHALL be retained only if it demonstrates observer-local relevance within the analysis window, for example:
   - it rises above the horizon, or
   - it reaches service visibility (`theta_min`), or
   - it contributes to the retained local-pass coverage objective.
4. If a budget cap is still required after observer-local filtering, the cap SHALL be applied using observer-local pass metrics rather than provider-wide even spacing.
5. The retained set SHALL preserve time staggering and azimuth diversity across local passes when the source geometry supports it.
6. Fixed sample caps such as `320` or `180` SHALL NOT remain hidden global defaults for observer-sky `real-trace` fixtures unless they are explicitly justified, documented, and validation-covered.

### 4.3 Startup Bootstrap

1. Startup bootstrap SHALL select an epoch within the replay window that improves initial observer-sky readability relative to simply using raw window start.
2. Bootstrap selection SHALL be based on physical geometry only, such as:
   - active/pass-visible count
   - high-elevation count
   - azimuth spread
   - observer-sky screen-space coverage proxy derived from the same physical pool
3. Bootstrap SHALL NOT rewrite satellite positions, inject satellites, or fabricate motion.
4. Startup bootstrap SHALL remain deterministic for the same fixture and configuration.

### 4.4 Replay Modes

1. The project SHALL distinguish two contracts:
   - `research-default real-trace`
   - `demo replay-window real-trace`
2. `research-default real-trace` SHALL remain the default path and SHALL use continuous propagation over the selected fixture epoch origin without forced looping.
3. `demo replay-window real-trace` MAY loop, but SHALL be clearly labeled as non-default presentation behavior.
4. A replay window intended for looped demo use SHOULD cover at least one representative orbital period for the retained constellation subset, unless a longer window is required to satisfy local-pass coverage objectives.
5. Looping SHALL NOT present as a silent mid-pass teleport. The seam SHALL either:
   - be made visually continuous by design, or
   - be made explicit as a replay reset boundary.

### 4.5 Research vs Demo Separation

1. A visibility-rich startup or loop-friendly replay window SHALL NOT be misrepresented as the generic research-default physical state of the live constellation.
2. Any mode that optimizes for persistent high-elevation readability or loop presentation SHALL be labeled and validation-covered as presentation/demo behavior.
3. Research-default outputs, manifests, and traceability artifacts SHALL indicate whether the run used:
   - continuous research mode, or
   - replay-window demo mode.

### 4.6 Frontend Acceptance Binding

1. The retained observer-local fixture pool SHALL support the observer-sky acceptance semantics defined in `docs/zh-TW/07-observer-sky-visual-acceptance.md`.
2. This package SHALL NOT solve observer-sky readability by faking screen-space jitter or display-only randomization.
3. If full-source geometry cannot support a requested "always crowded zenith" visual target, the implementation SHALL preserve physical truth and expose the limitation rather than inventing density.

## 5. Data Contract Target

Minimum future fixture metadata target:

```ts
interface RealTraceFixtureMetadata {
  provider: 'starlink' | 'oneweb';
  sourceSnapshotFile: string;
  sourceRecordCount: number;
  retainedRecordCount: number;
  replayWindowStartUtc: string;
  replayWindowDurationSec: number;
  bootstrapStartOffsetSec?: number;
  observer: {
    latDeg: number;
    lonDeg: number;
    altKm: number;
  };
  selectionPolicy: {
    mode: 'observer-local-pass';
    serviceMinElevationDeg: number;
    analysisWindowSec: number;
    rankingPolicyId: string;
  };
  replayModeSupport: {
    researchDefault: true;
    demoLoopSupported: boolean;
  };
}
```

Required rules:
1. fixture metadata SHALL describe the observer-local filtering basis.
2. the replay window SHALL be explicit in metadata.
3. bootstrap offset SHALL be explicit when enabled.
4. demo-loop support SHALL be explicit rather than inferred.

## 6. Intended Ownership Split

Future implementation SHOULD keep these responsibilities separate:
1. `scripts/sync-tle-fixtures.mjs`
   - raw provider snapshot ingestion
   - observer-local pass analysis
   - retained fixture emission and metadata
2. `src/sim/orbit/catalog.ts`
   - fixture parsing
   - replay-window metadata loading
3. `src/sim/scenarios/real-trace.ts`
   - epoch origin
   - startup bootstrap
   - research/default vs demo replay-mode selection
4. `src/viz/satellite/**`
   - continue to consume physical geometry pools only
   - do not own fixture-selection policy

The display layer SHALL NOT become the owner of TLE sampling policy.

## 7. Validation Gates (Pass/Fail)

1. Gate RTLP-1: epoch-origin correctness
   - the same fixture yields the same `t=0` geometry independent of wall-clock execution date.
2. Gate RTLP-2: local-pass retention relevance
   - the retained fixture demonstrates materially better NTPU local-pass coverage than constellation-wide even sampling at the same budget.
3. Gate RTLP-3: startup bootstrap readability
   - bootstrap start epoch produces a more observer-readable initial frame than raw replay-window start under deterministic scoring.
4. Gate RTLP-4: no fake density
   - all displayed satellites at startup and during replay are traced to retained real TLE records; no synthetic fill-in is allowed.
5. Gate RTLP-5: research/demo separation
   - manifests and runtime metadata identify whether replay-window demo mode is active.
6. Gate RTLP-6: loop-seam discipline
   - demo loop mode does not re-enter as a silent mid-pass teleport; reset/continuity behavior is explicit and deterministic.
7. Gate RTLP-7: stage safety
   - when implementation begins, `npm run lint`, `npm run test:sim`, `npm run build`, and `npm run validate:stage` pass with refreshed artifacts.

## 9. Open Questions

1. Should startup bootstrap maximize:
   - high-elevation count,
   - service-visible count,
   - screen-space spread,
   - or a weighted combination?
2. Should replay-window duration default to:
   - one orbital period estimate,
   - a fixed window length,
   - or provider/profile-specific policy?
3. What retained-record budget is sufficient for frontend responsiveness once observer-local pass filtering replaces even sampling?
4. Should demo loop mode reset actor memory explicitly at seam, or should it require a pre-roll overlap window for visual continuity?
