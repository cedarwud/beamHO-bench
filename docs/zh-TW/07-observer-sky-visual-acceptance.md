# 07. Observer-Sky 前台畫面驗收標準

## 1. 目的

這份文件只定義「前台畫面最後應該長什麼樣」與「什麼不算完成」。

它不是實作設計文件，也不是 SDD。
後續任何與衛星前台顯示、移動、observer-sky 視圖相關的 SDD，都必須引用這份文件，不能自行降低完成標準。

---

## 2. 前台畫面要表達的核心語意

1. 前台主要語意是「固定地面觀測點看天空中的 LEO 衛星通過」。
2. `NTPU` 地面場景是參考座標與研究背景，不是衛星視覺語意的主體。
3. 衛星在畫面上應表達 `rise -> pass -> set`，也就是：
   - 從某個地平線方向進場
   - 穿過天空中的可視弧段
   - 往另一側離場
4. 前台目的是研究模擬器，不是裝飾性飛行動畫。

---

## 3. 衛星應如何出現與消失

1. `elevation < 0 deg`
   - 不顯示
2. `0 deg <= elevation < theta_min`
   - 允許顯示，但必須是 `ghost` / 半透明 / 非服務中狀態
3. `elevation >= theta_min`
   - 顯示為 `active`
4. 畫面不可讓衛星像是憑空在天空中央生成或消失。
5. 即使有 `ghost` 區段，也不能把 `ghost` 誤表達成服務中衛星。

---

## 4. 衛星應如何移動

1. 同一顆衛星必須可被理解為同一個連續移動物體。
2. 移動應沿 observer-centric sky arc 進行，而不是像一批點在場景上空交換位置。
3. 衛星不需要全部經過正上方，但必須都在「天空半球」中沿自己的弧線移動。
4. 不可出現明顯的「停很久，接著突然跳到另一個定點」。
5. 若衛星離場，應該是因為真的接近地平線或超出顯示策略，而不是因為下一幀重選名單。

---

## 5. 時間與空間錯開

1. 同時可見的衛星，在時間上應有先後進場、先後離場的錯開。
2. 同時可見的衛星，在空間上應有不同方位角與不同仰角層次。
3. 不應該出現大量衛星長時間擠在畫面中央上方。
4. 不應該出現一批衛星幾乎同步、以同樣方式移動。
5. 這種錯開應由星座幾何、相位、observer geometry、visibility threshold 自然產生，不是純視覺亂數打散。

---

## 6. Display Set 與 Handover Candidate Set

1. 畫面中可見的衛星集合，不得直接等同於 HO candidate set。
2. 天空中可以看到的衛星，通常應多於真正參與換手決策的候選集合。
3. 調整 `candidateSatelliteLimit` 不應讓前台 observer-sky 畫面立刻縮成同樣數量的 top-N 衛星。
4. 若物理上存在更廣的 above-horizon 衛星池，前台應能保留比 HO candidate set 更完整的天空資訊。

---

## 7. 相機與視角驗收

1. 衛星前台的主要驗收視角，可以是 `上帝視角 / god-view`，不必是第一人稱地面抬頭視角。
2. 但這個主要視角必須仍然清楚表達 `observer-centric sky pass`，而不是目前校園第三人稱遠景中一團衛星擠在場景中央上方。
3. 若保留校園第三人稱遠景，它只能是輔助視角，不能作為 observer-sky 正確性的主要驗收依據。
4. 在主要驗收視角下，使用者必須能直觀看出：
   - 衛星從哪個方向進場
   - 哪些處於低仰角
   - 哪些已進入 `active` 可服務區
   - 哪些正在離場
5. 若在主要視角下仍然看起來像「一團在中央上方」，則視為不通過。

---

## 8. 三種模式的一致性

以下三種模式都必須遵守相同的 observer-sky 顯示原則：

1. `Synthetic Orbit`
2. `Starlink TLE`
3. `OneWeb TLE`

差異可以存在於：

1. 衛星密度
2. 通過頻率
3. 路徑方向
4. 可見數量

但不應存在於：

1. 顯示語意不同
2. 一種模式像真正的天空通過，另一種模式只是點群輪替

---

## 9. 不算完成的情況

只要出現以下任一情況，就不能宣告 observer-sky 前台完成：

1. 衛星大多仍集中在畫面中央上方。
2. 畫面仍主要呈現為一批衛星點在場景中心附近換位置。
3. 衛星的出現與消失看起來像憑空生成/消失，而不是從地平線附近進出。
4. 同一顆衛星無法被視為連續移動物體。
5. `candidateSatelliteLimit` 仍直接支配前台可見衛星集合。
6. 驗證只證明 `azimuth sector` 或 `determinism`，但沒有證明最終畫面行為。
7. 主要驗收視角仍只是校園第三人稱遠景，沒有專門為 observer-centric sky pass 設計的主視角。

---

## 10. 後續 SDD 必須回答的問題

任何新的前台相關 SDD，至少必須明確回答：

1. 主要驗收用的 god-view / observer-centric composition 是什麼
2. screen-space 上如何判定「不再團聚在中央」
3. 如何驗證衛星的進場、通過、離場可被辨識
4. 如何驗證時間與空間上的錯開
5. 如何保證 display set 與 candidate set 分離
6. 如何限制相鄰 tick 的 membership churn
7. 哪些指標是自動化驗證，哪些需要人工驗收

---

## 11. 驗收原則

1. 後續 SDD 不能只用代理指標結案。
2. `lint`、`test:sim`、`build`、`validate:stage` 通過，不代表前台畫面已符合這份文件。
3. 只有當自動化驗證與實際畫面都符合本文件時，observer-sky 前台才可視為完成。
